# 一、核心架构原则

## 1.1 唯一数据源

整个系统只有 **一处** 维护用例名和模块路径：`test_engine.py` 的 `CASE_REGISTRY` 和 `CASE_CN_NAMES`。

其他所有文件的映射表均**自动派生**，无需手动维护：

```
test_engine.CASE_REGISTRY   ← 英文名 → 模块路径（唯一数据源）
test_engine.CASE_CN_NAMES  ← 英文名 → 中文名（唯一数据源）
         ↓ 派生                        ↓ 派生
config_ui.CASE_NAME_MAP    report_generator.CASE_NAME_CN_MAP
config_ui._case_cn_to_en   report_generator.CASE_TO_CATEGORY
```

**新增用例只需修改 `test_engine.py` 中这两处**，其他文件自动同步。

# 二、数据流总览

```
┌─────────────────────────────────────────────────────────────┐
│                         config_ui.py                         │
│  1. _generate_test_conditions() → _all_conditions           │
│  2. _filter_conditions_by_case() → filtered_conditions        │
│  3. _build_test_engine_config() → config dict                │
└─────────────────────────────────────────────────────────────┘
                              ↓ config
┌─────────────────────────────────────────────────────────────┐
│                         test_engine.py                        │
│  TestEngine.run() → 对每个 case 调用 setup/execute/teardown  │
│  结果 → JSON 文件（summary + results + export_time）         │
└─────────────────────────────────────────────────────────────┘
                              ↓ JSON
┌─────────────────────────────────────────────────────────────┐
│                     report_generator.py                       │
│  generate_excel() → Excel 报告（汇总 + 用例sheets + 波形）   │
└─────────────────────────────────────────────────────────────┘
```

# 三、文件对应关系

| 文件 | 作用 | 维护方式 |
|------|------|----------|
| `test_cases/xxx_tests/XXXTest.py` | 测试用例主体代码 | 手动创建 |
| `test_engine.py` → `CASE_REGISTRY` | 注册：英文名 → 模块路径 | **只需改这里** |
| `test_engine.py` → `CASE_CN_NAMES` | 注册：英文名 → 中文名 | **只需改这里** |
| `test_engine.py` → `_apply_xxx_config()` | 参数注入（如需特殊参数） | 手动添加 |
| 测试用例 → `COLS` | 报告列定义（含列宽） | 手动添加 |
| `config_ui.py` → `_filter_conditions_by_case()` | 条件筛选逻辑（如需） | 手动添加 |

> **报告生成器不维护列定义**。列定义在测试用例的 `COLS` 中，报告生成器只负责读取和渲染。

# 五、测试用例模板

## 5.1 通用结构

```
setup()           ← 仪器初始化，从 params 读取 UI 配置的通道/衰减参数
    ↓
execute()         ← 主流程编排
    ↓
teardown()        ← 统一为 _step_discharge()（AC OFF → eload CC 1A ON → wait 1s → OFF）
```

## 5.2 execute() 标准流程

```python
def execute(self, instruments):
    ac   = instruments.get("AC_SOURCE")
    elod = instruments.get("ELOAD")
    osc  = instruments.get("OSC")
    snif = instruments.get("SNIFFER")

    for input_cond, test_cond in self._iter_conditions():
        # 1. 开机自检
        if not self.startup_self_check(instruments, input_cond):
            self._step_power_off(ac, elod)
            self.sub_results.append(self._make_result(..., skipped=True, ...))
            continue

        # 2. 诱骗器协议
        sniffer_ok = self._step_setup_sniffer(snif, proto_label, vout, iout)

        # 3. 电子负载上电
        self._step_setup_eload(elod, iout_target)

        # 4. 抓取波形/测量
        self._step_capture_and_measure(osc, ch_out, ...)

        # 5. 判定 & 记录结果
        self.sub_results.append(self._make_result(..., overall_pass=..., ...))

        # 6. 放电（下电）
        self._step_discharge(ac, elod)
```

## 5.3 setup() 模板

```python
def setup(self, instruments: Dict[str, Any]):
    self.sub_results = []
    super().setup(instruments)   # ← 必须调用，内部调用 startup_self_check
```

**禁止省略 `super().setup(instruments)`**。

## 5.4 teardown() 模板

```python
def teardown(self, instruments: Dict[str, Any]):
    self._step_discharge(
        instruments.get("AC_SOURCE"),
        instruments.get("ELOAD"),
    )
    osc = instruments.get("OSC")
    if osc:
        try:
            osc.set_timebase_mode("MAIN")   # ← 恢复示波器普通模式
        except Exception:
            pass
```

## 5.5 _make_result() 模板

**核心规则：返回的 key 必须与 COLS 列头完全一致。**

```python
def _make_result(self, *,
                 input_cond: str,
                 proto_label: str,
                 vout_target: float,
                 iout_target: float,
                 waveform: str = None,
                 sniffer_ok: bool,
                 overall_pass: bool,
                 fail_reason: str = "",
                 skipped: bool = False,
                 **extra) -> dict:
    """组装单条测试结果。"""
    return {
        # 报告字段（必须与 COLS 列头一致）
        "输入条件":   input_cond,
        "协议":       proto_label,
        "输出电压(V)": vout_target,
        "输出电流(A)": iout_target,
        "测试波形":   waveform,
        "测试结论":   "SKIP" if skipped else ("PASS" if overall_pass else "FAIL"),
        "备注":       fail_reason,
        # 内部字段（不入报告）
        "overall_pass": overall_pass,
        "fail_reason":  fail_reason,
        "skipped":      skipped,
        **extra,
    }
```

# 六、COLS 体系（报告列定义）

## 6.1 规范

**每个测试用例在自己的 COLS 中定义报告列顺序和列宽，报告生成器只负责读取和渲染。**

> ⚠️ **`COLS 中不能定义前缀列（序号/用例名称/测试结论）`**
> 这三列由 `report_generator._flatten()` 统一注入，`_write_case_sheet` 作为前缀列渲染。
> 如果在 COLS 中重复定义，会导致 `all_render_keys` 计数错误，
> `prefix_len` 与实际前缀数量不匹配，合并格列位置全错，效率表格数据与列头完全错位。

```python
class XXXTest(TestCase):
    # 注意：序号 / 用例名称 / 测试结论 不在这里定义！
    COLS = [
        ("输入条件",       16),
        ("协议",           12),
        ("输出电压(V)",   13),
        ("输出电流(A)",   13),
        ("负载点",          9),
        ("纹波要求",       12),
        ("纹波实测值(mV)", 16),
        ("测试波形",       18),
        ("备注",          32),
    ]
```

## 6.2 列宽定义位置

列宽只在测试用例的 `COLS` 中定义，**报告生成器不重复定义列宽**。

## 6.3 列头与字段名必须一致

**核心规则**：`COLS 列头 = sub_result 的 key 名`（两者必须完全一致）

报告生成器用列头名直接去 JSON 查值，不需要任何 alias 转换。

# 七、内部字段约定

以下字段是内部逻辑用的，**不入 Excel 报告**：

| 字段 | 用途 | 报告处理 |
|------|------|---------|
| `overall_pass` | 整体 PASS 判定 | ❌ 不入报告 |
| `fail_reason` | 失败原因 | ✅ 写入"备注"列 |
| `skipped` | 跳过标记 | ❌ 不入报告 |

以下标准报告字段由内部字段构造：

| 报告字段 | 构造方式 |
|---------|---------|
| `测试结论` | `"SKIP" if skipped else ("PASS" if overall_pass else "FAIL")` |
| `备注` | `fail_reason` |

# 八、base.py 公共方法

## 8.1 仪器获取 Helper（推荐使用）

```python
osc   = self._osc(instruments)      # 替代 instruments.get("OSC")
eload = self._eload(instruments)    # 替代 instruments.get("ELOAD")
ac    = self._ac(instruments)      # 替代 instruments.get("AC_SOURCE")
dc    = self._dc(instruments)      # 替代 instruments.get("DC_SOURCE")
pwr   = self._pwrmeter(instruments) # 替代 instruments.get("POWER_METER")
snif  = self._sniffer(instruments)  # 替代 instruments.get("SNIFFER")
```
未连接时自动记录警告，无需手动 `if inst is None` 判断。

## 8.2 步骤方法

| 方法 | 说明 |
|------|------|
| `_step_power_off(ac, elod)` | 仅关输出，用于 startup 失败时快速下电 |
| `_step_discharge(ac, elod, current=1.0, duration=1.0)` | 放电：AC OFF → eload CC 1A ON → wait → OFF |
| `_step_setup_sniffer(sniffer, proto_label, vout, iout)` | 诱骗器协议配置 |
| `_step_setup_eload(elod, iout)` | 电子负载 CC 模式上电 |
| `_step_capture_and_measure(osc, ch_out, ...)` | 通用波形保存（含 `_save_waveform`） |
| `_get_waveform_dir()` | 获取波形保存目录（来自 `params["osc_waveform_dir"]`） |
| `startup_self_check(instruments, input_cond)` | 开机自检（从 `self.params` 读取配置） |

# 九、仪器 Key 名称规范（重要）

`instrument_manager.py` 创建仪器实例时使用的 key **不是** 类别全称，而是固定的短名：

| 实际 key（代码中必须用这个） | config_ui 类别名 | 驱动 |
|---|---|---|
| `"OSC"` | `oscilloscope` | DSOX4024A |
| `"ELOAD"` | `electronic_load` | IT8511 |
| `"AC_SOURCE"` | `ac_source` | IT7321 |
| `"POWER_METER"` | `power_meter` | WT333E |
| `"SNIFFER"` | `sniffer` | IP2716Sniffer |
| `"DC_SOURCE"` | `dc_source` | IT6333A |

```python
# ✅ 正确
osc = instruments.get("OSC")

# ❌ 错误（osc 永远是 None）
osc = instruments.get("OSCILLOSCOPE")
```

# 十、DSOX4000A SINGLE 触发（重要）

DSOX4000A 示波器的单次触发正确命令是 `:SINGLE`，驱动封装为 `set_single_trigger()` 方法。

**配置顺序必须正确**：

```python
# ✅ 正确
osc.set_trigger_source("CHAN2")
osc.set_trigger_coupling("DC")
osc.set_trigger_slope("BOTH")
osc.set_trigger_level(vout * 0.5)
osc.set_single_trigger()    # ← 最后武装
time.sleep(2.0)            # ← 等待进入 ARM 等待状态

# ❌ 错误
osc.set_trigger_mode("SINGLE")   # DSOX4000A 不支持
```

# 十一、UI 参数来源（必须从 self.params 读取）

所有仪器相关参数**必须从 `self.params`** 读取，禁止硬编码。

| 参数 | 来源 | 说明 |
|------|------|------|
| 示波器输入通道 | `params["osc_input_ch"]` | 默认 "CH4"，UI 配置 |
| 示波器输出通道 | `params["osc_output_ch"]` | 默认 "CH2"，UI 配置 |
| 示波器输入衰减 | `params["osc_input_attn"]` | 默认 1.0，UI 配置 |
| 示波器输出衰减 | `params["osc_output_attn"]` | 默认 1.0，UI 配置 |
| 功率计输入电流通道 | `params["pwr_in_i_ch"]` | 默认 "CH1"，UI 配置 |
| 波形保存目录 | `params["osc_waveform_dir"]` | TestEngine 自动注入 |
| 测试条件 | `params["test_conditions"]` | TestEngine 从 filtered_conditions 注入 |

**通道解析 helper**：
```python
def _parse_pwr_ch(self, ch_str: str) -> int:
    """\"CH1\" → 0, \"CH2\" → 1"""
    try:
        return int(ch_str.replace("CH", "").replace("ch", "")) - 1
    except Exception:
        return 0
```

# 十五、常犯错误汇总

### 错误 1：CASE_REGISTRY 路径写错
- **现象**：引擎启动时 `_validate_case_registry()` 报警告
- **排查**：确认模块路径格式为 `test_cases.xxx_tests.XXXTest`，且文件真实存在

### 错误 2：仪器 key 用错了名字
- **现象**：日志中 `[XXX] 示波器未连接` 警告，但示波器明明连着
- **根因**：`instruments.get("OSCILLOSCOPE")` 永远是 `None`（正确 key 是 `"OSC"`）

### 错误 3：DSOX4000A SINGLE 触发命令错误
- **现象**：示波器不进入等待触发状态，触发永远等不到而超时
- **正确做法**：用 `osc.set_single_trigger()` 方法

### 错误 4：filtered_conditions 用中文 key
- **现象**：点击 case 名称后条件列表为空
- **排查**：`filtered_conditions` key 必须是英文用例名

### 错误 5：仪器参数未从 params 读取
- **现象**：示波器通道/功率计通道用硬编码，UI 配置不生效
- **正确做法**：`ch = int(self.params.get("osc_output_ch", 2))`

### 错误 6：teardown 未恢复示波器时基模式
- **现象**：下一次测试示波器仍在 ROLL 模式
- **正确做法**：teardown 中调用 `osc.set_timebase_mode("MAIN")`

### 错误 7：`_apply_xxx_config` 只写在了一处
- **现象**：首次加载 0 条测试条件，只有克隆的新实例才有
- **排查**：确认在 `_load_single_case` **和** `_create_fresh_instance` 两处都有调用

### 错误 8：sub_result 字段名与 COLS 列头不一致
- **现象**：报告该列数据为空
- **排查**：确认 `_make_result` 返回的 key == `COLS` 中的列头

### 错误 9：漏掉必要的报告字段
- **现象**：报告缺少"测试结论"或"备注"列
- **正确做法**：确保 `_make_result` 返回 `测试结论` 和 `备注` 字段

### 错误 10：DSOX4024A 过冲/负冲测量未过滤无效值
- **现象**：过冲/负冲始终为 0 或极大值（+9.9E+37）
- **根因**：`measure_overshoot_builtin` / `measure_undershoot_builtin` 直接用 `float(resp)`，未调用 `_parse_measurement()` 过滤 Keysight 的无效 sentinel 值
- **正确做法**：始终用 `self._parse_measurement(resp)` 包裹查询结果

### 错误 11：PowerOnOffTest 测量过冲/负冲前未 STOP 示波器
- **现象**：过冲/负冲数据始终为 0，波形中无测量值
- **根因**：DSOX4024A 的 `:MEAS:VOVER?` / `:MEAS:VUND?` 必须在 STOP 状态下才能返回有效值
- **正确做法**：触发等待完成后、`get_measurement` 前必须调用 `osc.stop()` 冻结采集

### 错误 12：PowerOnOffTest 触发轮询间隔过粗
- **现象**：关机波形经常触发超时，测不到数据
- **根因**：`TRIGGER_WAIT_S = 2.0` 秒，关机瞬态 ~100ms，5秒内最多查3次，极易错过
- **正确做法**：`TRIGGER_WAIT_S` 设为 0.3 秒，5秒内可轮询约17次

### 错误 13：InputDipTest `_make_result` 参数名错误
- **现象**：测试报 ERROR：`got an unexpected keyword argument '最大值'`
- **根因**：调用处用中文 `最大值=` 作为关键字参数，但 `_make_result` 签名用英文 `osc_vmax=`
- **正确做法**：关键字参数必须与函数签名完全一致，例如 `osc_vmax=osc_vmax`

### 错误 14：InputDipTest 波形路径未传入 `_make_result`
- **现象**：JSON 中 `测试波形` 字段为空，Excel 报告看不到波形链接
- **根因**：`wave_path = _step_capture_and_measure(...)` 捕获了路径但未传递给 `_make_result`
- **正确做法**：`waveform` 作为 `_make_result` 的显式参数传入，签名声明为 `waveform: str = ""`

### 错误 15：report_generator `_display_name` 重复加"测试"后缀
- **现象**：Excel Sheet 名称和表头变成"输入测试-测试明细"而非正确的用例名
- **根因**：`CASE_CN_NAMES` 的中文名已含"测试"，`_display_name` 又强行追加一遍，导致 `输入跌落测试` → `输入跌落测试测试` → 分类前缀匹配变成 `"输入测试"`
- **正确做法**：中文名直接从 `CASE_CN_NAMES` 取，不重复加后缀

### 错误 16：report_generator `_eff_alias` 错误映射导致 EfficiencyTest 列数据为空
- **现象**：能效测试报告的"协议"、"输出电压(V)"、"输出电流(A)" 列始终为空
- **根因**：错误的 alias 映射把列头名映射到不存在的 key（如 `协议` → `proto_label`），实际 JSON 存的 key 是 `协议`
- **正确做法**：删除 `_eff_alias`，所有用例统一用列头名直接作为 JSON key 查询

### 错误 17：`_fix_rels` 导致 Excel 文件损坏
- **现象**：Excel 打开时报"此工作簿的某些部分可能已被修复或丢弃"
- **根因**：`_fix_rels` 修改 ZIP 时使用 `ZIP_DEFLATED` 覆盖原始压缩类型，破坏文件结构
- **正确做法**：移除 `_fix_rels` 调用（openpyxl 3.x 自身保存的 rels 已基本正确）

from openpyxl.worksheet.hyperlink import Hyperlink

hl = Hyperlink(

# 十七、Debug 经验（问题案例）

### 经验 1：工作区文件不能 shadow Python 标准库

**问题**：numpy 导入失败，报 `AttributeError: module 'inspect' has no attribute 'cleandoc'`

**根因**：工作区存在 `inspect.py`，shadow 了 Python 标准库的 `inspect` 模块。

**解决**：删除工作区的 `inspect.py` 及任何与标准库同名的文件。

### 经验 2：_key_alias 已删除

早期版本的报告生成器有 `_common_alias` 做字段名映射，但映射逻辑是错的（把中文 key 映射到英文 key，但实际 JSON 存的是中文 key）。

**当前架构**：不再需要 alias。`_make_result` 返回的 key 必须与 `COLS` 列头完全一致，报告生成器直接用列头名查 JSON。

### 经验 3：_fix_rels 支持新文件写入

早期版本的 `_fix_rels` 只处理 ZIP 中已有的 rels 文件，不支持写入新文件（如 `sheet3.xml.rels`）。

**当前版本**：使用 `writestr` 写入模式，支持创建原 ZIP 中不存在的新文件。
